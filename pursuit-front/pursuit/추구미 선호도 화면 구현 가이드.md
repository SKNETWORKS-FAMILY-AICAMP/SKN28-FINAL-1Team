# 추구미 선호도 화면 구현 가이드

## 최종 화면 구현 방향

최종 화면은 `local/final_project/pursuit_preference_ui_avoided_mode (1).png` 이미지와 같은 구조로 구현한다.

- 상단에 `선호해요` / `피하고 싶어요` 모드 토글을 둔다.
- 사용자는 현재 모드에 따라 같은 옵션을 선호 또는 기피로 선택한다.
- 선택은 모두 optional이다. 아무것도 선택하지 않아도 저장할 수 있어야 한다.
- 저장된 값은 다시 화면에 진입했을 때 Blue/Red 선택 상태로 조회되어야 한다.
- 사용자는 이후 같은 화면에서 다시 수정할 수 있어야 한다.
- 수정은 별도 API가 아니라 저장 API `PUT`으로 전체 상태를 다시 보내 덮어쓴다.

이 구조에서는 백엔드 저장값도 `preferred_styles`, `avoided_colors`처럼 필드를 납작하게 많이 두기보다, 아래처럼 `preferred`와 `avoided`를 나누고 그 안에 카테고리별 배열을 두는 방식이 화면과 가장 잘 맞는다.

```json
{
  "preferred": {
    "seasons": ["spring"],
    "styles": ["minimal", "lovely", "retro"],
    "colors": ["black", "white", "blue"],
    "pants_fits": ["wide"],
    "skirt_lengths": ["mini"],
    "skirt_types": ["a_line"],
    "necklines": ["round"],
    "sleeves": ["long"],
    "occasions": ["daily"]
  },
  "avoided": {
    "seasons": [],
    "styles": [],
    "colors": ["neon", "pink", "fluorescent"],
    "pants_fits": ["skinny"],
    "skirt_lengths": [],
    "skirt_types": [],
    "necklines": [],
    "sleeves": [],
    "occasions": []
  }
}
```

## 개요

본 문서는 `PursuitPreferenceScreen.tsx` 컴포넌트의 사용 방법 및 API 연동 방법을 설명합니다.

## 파일 구조

```
PursuitPreferenceScreen.tsx
├── Types & Interfaces
│   ├── PreferenceMode: 'preferred' | 'avoided'
│   ├── PreferenceOption: 각 옵션의 메타데이터
│   ├── CategoryOptions: 전체 카테고리별 옵션 목록
│   └── UserPreferences: 사용자 선호도 상태
├── Mock Data (MOCK_CATEGORY_OPTIONS)
├── Chip Component: 개별 선택 칩
├── CategorySection Component: 카테고리별 칩 그룹
└── PursuitPreferenceScreen Component: 메인 화면
```

## 핵심 기능

### 1. 토글 모드 관리
- `currentMode` 상태로 '선호해요' 또는 '피하고 싶어요' 모드를 관리합니다.
- 토글 버튼 클릭 시 모드만 변경되며, 기존 선택 상태는 유지됩니다.

### 2. 최신 선택 우선 로직
```typescript
// 예시: '그레이' 색상을 선호에서 선택 후 기피에서 선택
// 결과: '그레이'는 기피 목록에만 포함됨

if (currentMode === 'preferred') {
  newPrefs.preferred[categoryKey] = [...preferredList, code];
  newPrefs.avoided[categoryKey] = avoidedList.filter(c => c !== code);
} else {
  newPrefs.avoided[categoryKey] = [...avoidedList, code];
  newPrefs.preferred[categoryKey] = preferredList.filter(c => c !== code);
}
```

### 3. 시각적 상태 표현
- **선호 선택**: Blue (#007AFF) 배경
- **기피 선택**: Red (#FF3B30) 배경
- **미선택**: White 배경 + Gray 테두리

### 4. 색상 및 아이콘 표시
- **색상 옵션**: 칩 내부에 작은 컬러 박스 표시
- **핏/기장/넥라인 옵션**: 이모지 아이콘 표시

## API 연동

### 1. 옵션 목록 조회 (초기 로드)

프론트가 칩 목록을 하드코딩하지 않도록 백엔드에서 카테고리별 옵션을 내려준다.

```http
GET /api/v1/preference-options/
```

응답은 화면 카테고리 순서대로 내려주는 것이 좋다.

```json
{
  "categories": [
    {
      "key": "seasons",
      "label": "계절",
      "options": [
        { "code": "spring", "label": "봄" },
        { "code": "summer", "label": "여름" }
      ]
    },
    {
      "key": "styles",
      "label": "스타일",
      "options": [
        { "code": "minimal", "label": "미니멀" },
        { "code": "casual", "label": "캐주얼" }
      ]
    }
  ]
}
```

### 3. 사용자 선호도 저장/수정

저장과 수정은 같은 API를 사용한다.

```http
PUT /api/v1/users/me/style-preferences/
```

현재 화면의 전체 선택 상태를 그대로 보낸다. 누락된 카테고리는 백엔드에서 빈 배열로 처리하거나, 프론트에서 항상 빈 배열까지 포함해 보내는 방식으로 통일한다.

```typescript
// TODO: handleSave 함수 내 API 호출 추가

const handleSave = async () => {

  try {
    const response = await fetch('/api/v1/users/me/style-preferences/', {

        'Content-Type': 'application/json',

      },
      body: JSON.stringify(preferences),
    });
    
    if (response.ok) {
      Alert.alert('성공', '저장되었습니다.');
      // 이전 화면으로 네비게이션
    } else {
      throw new Error('Save failed');
    }
  } catch (error) {
    Alert.alert('오류', '저장 중 오류가 발생했습니다.');
  } finally {
    setIsLoading(false);
  }
};
```

## 문제 해결

### 칩이 렌더링되지 않음
- MOCK_CATEGORY_OPTIONS 데이터 확인
- CategorySection 컴포넌트 props 확인

### 저장이 되지 않음
- API 엔드포인트 URL 확인
- 인증 토큰 확인
- 요청 바디 형식 확인

### 성능 문제
- 옵션 개수가 많은 경우 FlatList로 변경
- 메모이제이션 추가 (React.memo)
