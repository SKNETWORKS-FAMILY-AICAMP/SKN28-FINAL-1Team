# 데이터셋 분석 아티팩트 작성 지침

이 문서는 `data/<데이터셋 폴더>/README.md` 분석 내용을 바탕으로 Claude Code Artifact 또는 로컬 `artifact.html`을 만들 때 쓰는 단일 지침이다.

아티팩트의 목적은 **데이터의 구조, 데이터 형태, 주요 변수, 실제 샘플을 한눈에 확인하는 것**이다. 추천 모델 구조, RAG 아키텍처, 파이프라인 설계, SQL 설계는 다루지 않는다.

## 기본 원칙

1. 실제 확인한 데이터만 쓴다.
   - `data/<데이터셋 폴더>/README.md`의 실측 근거를 우선 사용한다.
   - README에 없는 숫자나 분포가 필요하면 S3(`s3://skn28-cozy/...`)에서 직접 샘플링하거나 집계한다.
   - 개수, 용량, 분포, 컬럼명, 샘플 내용은 절대 추정해서 쓰지 않는다.

2. 한국어로 설명한다.
   - 기본 언어는 한국어다.
   - 영어 변수명, 필드명, 컬럼명은 반드시 `영어(한글)` 형식으로 병기한다.
   - 예: `article_id`(상품 ID), `semantic_category`(의미 카테고리), `customer_id`(고객 ID)

3. 데이터 분석 결과를 보여주는 데 집중한다.
   - 포함: 데이터 개수, 폴더 구조, 파일 형식, 변수 설명, RAG 활용 포인트, 실제 샘플, 이미지/대화/인물 예시, 데이터별 주의사항
   - 제외: RAG 아키텍처, 벡터DB 구조, API 설계, 서비스 화면 설계, SQL 스키마, 모델 학습 파이프라인

4. 맨 위 요약은 유지한다.
   - 데이터가 몇 개인지, 어떤 파일/폴더로 구성되는지, 핵심 형태가 무엇인지 보여주는 상단 요약은 반드시 유지한다.
   - 상단 통계 타일은 4~6개 정도로 제한한다.

## 아티팩트 구성 순서

아티팩트는 아래 순서를 따른다. 데이터셋 성격상 해당되지 않는 항목은 생략하되, 순서는 바꾸지 않는다.

### 1. 상단 요약

가장 먼저 데이터셋 규모와 핵심 형태를 보여준다.

포함 예시:
- 전체 데이터셋 수 또는 오브젝트 수
- 총 용량
- 핵심 파일 수
- 이미지/영상/텍스트/대화/3D/CSV/JSON 등 데이터 형태
- 주요 카테고리 수
- 샘플 또는 인물 수

표현 방식:
- 통계 타일 4~6개
- 짧은 한 문장 설명
- 실측 근거가 있으면 작은 note로 표시

### 2. 데이터 구조와 데이터 형태

데이터가 어떤 폴더와 파일로 구성되는지 보여준다.

포함할 내용:
- 전체 폴더 구조 요약
- 핵심 파일 또는 폴더의 역할
- 파일 형식: CSV, JSON, Parquet, 이미지, 영상, 3D mesh, 라벨 파일 등
- 각 파일이 서로 연결되는 방식

표현 방식:
- 폴더 트리 또는 구조 카드
- 핵심 파일 표
- 파일 간 관계가 중요하면 간단한 연결 다이어그램

금지:
- SQL 테이블처럼 재설계하지 않는다.
- 데이터베이스 스키마 설명으로 바꾸지 않는다.

### 3. 변수와 필드 설명

데이터별 변수명, 컬럼명, 필드명이 있으면 반드시 설명한다.

작성 규칙:
- 변수명은 `영어(한글)` 형식으로 쓴다.
- 의미, 값의 형태, 예시 값을 같이 보여준다.
- 중요한 변수부터 먼저 보여준다.
- 변수가 많으면 핵심 변수와 보조 변수로 나눈다.

표 예시:

| 변수 | 의미 | 값 예시 |
|---|---|---|
| `article_id`(상품 ID) | 상품을 구분하는 고유 ID | `706016001` |
| `product_type_name`(상품 유형명) | 상품의 세부 유형 | `Dress` |
| `colour_group_name`(색상 그룹명) | 상품 색상 그룹 | `Black` |

### 4. RAG 활용 포인트

이 데이터가 RAG에서 어떻게 쓰이면 좋은지 간단히 설명한다.

작성 방향:
- 데이터 자체의 활용 포인트만 쓴다.
- 검색 질문, 검색 기준, 답변 근거로 쓸 수 있는 필드를 연결한다.
- 아키텍처, 벡터DB, 임베딩 파이프라인 설명은 쓰지 않는다.

좋은 예:
- 상품 메타데이터는 `product_type_name`(상품 유형명), `colour_group_name`(색상 그룹명), `detail_desc`(상세 설명)를 함께 묶어 상품 검색 근거로 쓸 수 있다.
- 대화 데이터는 사용자 요청 표현과 추천 응답의 실제 문장 패턴을 보여주는 예시 문서로 쓸 수 있다.
- 신체 치수 데이터는 `height`(키), `waist`(허리둘레), `chest`(가슴둘레) 같은 실측 기준을 사이즈 추천 근거로 쓸 수 있다.

나쁜 예:
- FAISS, Pinecone, Chroma 같은 저장소 구조 설명
- Retriever-Generator 아키텍처 설명
- LangChain 체인 설계
- API 라우팅 또는 서비스 배포 구조

### 5. 샘플 아이템/대화/인물

해당되는 샘플을 실제 데이터에서 1~3개 보여준다.

데이터별 예시:
- 상품 데이터: 샘플 상품 카드, 상품명, 카테고리, 색상, 설명, 이미지
- 코디 데이터: 코디 세트, 포함 아이템, 호환성 라벨, 이미지
- 대화 데이터: 실제 대화 1개를 채팅 UI로 표시
- 인물/신체 데이터: 샘플 인물 1명, 성별/연령대/치수/3D 또는 2D 이미지
- 이미지/영상 데이터: 대표 이미지 또는 영상 프레임
- 룰 데이터: 샘플 룰 2~3개와 적용 조건

규칙:
- 실제 샘플만 사용한다.
- 샘플 ID, 파일명, 근거 파일을 함께 표시한다.
- 개인정보나 민감 정보가 있으면 식별 불가능한 범위로만 보여준다.

### 6. 이미지로 보여줄 수 있는 것

이미지, 프레임, 썸네일, 3D 캡처, 라벨 시각화가 가능하면 반드시 보여준다.

우선순위:
1. 실제 이미지 파일을 base64 또는 접근 가능한 경로로 표시
2. 영상 데이터는 대표 프레임 또는 썸네일 표시
3. 3D 데이터는 캡처 이미지나 구조 다이어그램 표시
4. 실제 이미지 표시가 어렵다면 파일 구조와 샘플 메타데이터를 카드로 표시

주의:
- 이미지가 없는데 있는 것처럼 만들지 않는다.
- 샘플 이미지를 표시했다면 어떤 파일에서 온 이미지인지 적는다.

### 7. 데이터별 필요한 설명

마지막에는 해당 데이터셋을 볼 때 꼭 알아야 하는 설명만 덧붙인다.

포함할 수 있는 내용:
- 라벨 해석 방법
- 파일명이 의미하는 규칙
- 데이터 누락 또는 편향
- 샘플링 시 주의점
- 단위: cm, mm, kg, pixel 등
- 대화/추천/코디 데이터의 해석상 주의사항

금지:
- 프로젝트 전체 아키텍처 설명
- 모델링 전략 장문 설명
- 별도 서비스 기획

## 시각화 기준

1. 한 화면에서 구조가 잘 보여야 한다.
   - 상단 요약 -> 구조 -> 변수 -> RAG 활용 -> 샘플 -> 이미지 -> 설명 순서가 자연스럽게 이어져야 한다.

2. 차트는 필요한 경우에만 쓴다.
   - 분포를 실제로 집계한 경우에만 막대그래프나 비율 차트를 사용한다.
   - 집계하지 않은 값은 차트로 만들지 않는다.

3. 디자인은 단정하게 유지한다.
   - 기존 `11_PoC_패션코디` 아티팩트의 깔끔한 분석 보고서 느낌은 유지한다.
   - 데이터셋마다 색상과 장식에 힘을 주기보다, 구조와 샘플을 읽기 쉽게 만든다.
   - 라이트/다크 테마와 모바일 화면에서 깨지지 않게 한다.

## 체크리스트

Artifact 작성 전후로 아래만 확인한다.

1. 맨 위에 데이터 개수/규모 요약이 있는가?
2. 데이터 구조와 데이터 형태가 먼저 보이는가?
3. 변수명이 모두 `영어(한글)` 형식인가?
4. RAG 활용 설명이 아키텍처가 아니라 데이터 활용 포인트인가?
5. 실제 샘플 아이템/대화/인물 중 해당되는 것이 들어갔는가?
6. 이미지로 보여줄 수 있는 데이터는 실제 이미지로 표시했는가?
7. 데이터별 주의사항이 마지막에 간단히 정리됐는가?
8. SQL, 벡터DB, 서비스 아키텍처 설명이 들어가지 않았는가?
