## B. 섹션별 작성 지침

### 1. 이 데이터는 무엇인가
- 깔끔한 **산문**으로 데이터의 정체를 먼저 설명한다.
- 핵심 자산은 `-` 불릿 또는 표로 정리한다.
- 데이터 형태(텍스트 / 이미지 / 분류 목록 / 영상 등)를 **명확히 분류·설명**한다.
- 구 11번 "규모와 주의사항"(총 용량, 오브젝트 수, 용량 분포, 주의사항)을 **표 형태로 이 섹션에 병합**한다.

예시(규모·주의사항 표):

| 항목 | 값 / 주의 |
|---|---|
| 총 용량 | 약 4.30GB (오브젝트 39개) |
| 가장 큰 파일 | `nondisjoint/train.parquet` 약 1.79GB |
| 주의 | `polyvore_item_metadata.json`은 한 줄 dict — `json.load` 시 메모리 큼, 스트리밍 파서 권장 |

### 2. 전체 폴더 구조
- 트리를 **하나로 통합**해 전체 구조를 한 번에 보여준다.
- 폴더 안 폴더·파일을 서브태스크별로 쪼개 **여러 트리로 나누지 말 것**.
- 코드블록(```text) 안에 트리 하나만 둔다.

### 3. 폴더별 의미
- 폴더·파일명은 **반드시 백틱**으로 구분한다 (예: `train.json`).
- 3.1, 3.2처럼 **세부 번호**를 부여한다.
- CSV 컬럼·변수 이름이 있으면 **표로 정리**해 설명한다.
- 영어 변수는 항상 **영어(한글) 병기** (예: `semantic_category`(대분류)).

컬럼 표 예시:

| 컬럼 | 의미 |
|---|---|
| `set_id` | 코디의 PK(기본키) |
| `items[].item_id` | 각 아이템의 ID(식별자) |
| `items[].index` | 코디 내 슬롯 순서(순번) |

### 4. 가장 중요한 파일
- 이 데이터셋에서 중심축이 되는 단일 파일을 지목하고, 실제 레코드 발췌와 핵심 키를 설명한다.
- 어떻게 다른 파일과 조인·연결되는지 서술한다.

### 5. 이 데이터로 알 수 있는 것
- **2열 표**로 작성한다.
- 왼쪽: 짧은 키워드(예: 순서, 빈도, 추천 이유).
- 오른쪽: 그에 해당하는 긴 설명 + 근거 파일/변수.

표 예시(그대로 참고):

| 키워드 | 설명 (근거 파일/변수) |
|---|---|
| 순서 | 코디봇이 어떤 순서로 사용자에게 질문하는지 (ASK_LENGTH → ASK_COLOR → SUGGEST_TYPE …) |
| 빈도 | 어떤 카테고리 조합이 자주 등장하는지 (`train.json` × `semantic_category` 집계) |
| 추천 이유 | 왜 이 아이템이 어울리는지의 근거 (`compatibility_*.txt` 라벨, `description`의 색상·소재) |

### 6. 추천 시스템에 활용하는 방법
- 활용 방안을 서술하되, **근거(파일·변수 인용)를 제시**한다.
- 구 9번보다 **간결하게** 정리한다. 소제목 나열은 짧게.

### 7. 샘플 보는 순서 + 최소 분석 루틴
- 구 6 "샘플 하나를 보는 순서"와 구 10 "초보자용 최소 분석 루틴"은 내용이 겹치므로 **하나로 병합**한다.
- 단계별 번호 목록 하나 + (선택) 흐름 다이어그램 한 개로 정리한다. 같은 절차를 두 번 쓰지 않는다.

### 8. 내려받은 샘플
- 이번 세션에서 실제로 내려받은 파일 목록과 그로부터 확인한 사실을 적는다.
- 내려받지 않은 것은 솔직히 "미확인"으로 명시한다.

### 9. 출처 및 확인 근거
- 출처·확인 근거는 **반드시 그 파일이나 폴더 안에서 실측 확인한 것만** 기재한다.
- Confluence 등 **외부 설명 재인용 금지** — 실물 검증 필수.
- "무엇으로(명령·파일) 무엇을 확인했다"는 형식으로 근거를 남긴다.

## C. 공통 금지 / 필수 규칙 (README·아티팩트 공통)

**금지**
- `**"텍스트"**`처럼 볼드와 따옴표 중첩 금지 — 마크다운이 볼드 처리 못함. `"텍스트"` 또는 `**텍스트**` 중 하나만 사용.
- SQL 관련 내용 전면 금지 (구 7번 섹션 삭제).
- 일본어·중국어 등 타 언어 혼입 금지.
- 외부(Confluence 등) 설명을 실측 없이 재인용 금지.

**필수**
- 기본 언어 한국어.
- 폴더·파일명은 **백틱** 표기 (예: `data/disjoint/`).
- 영어 변수·필드명은 항상 **영어(한글) 병기** — 아티팩트 HTML에도 동일 적용 (예: `category_id`(카테고리 ID)).

## D. 좋은 예 / 나쁜 예

**볼드·따옴표 중첩**

- 나쁜 예: `이 파일은 **"중심 테이블"**이다.` → 마크다운에서 볼드가 깨진다.
- 좋은 예: `이 파일은 **중심 테이블**이다.` 또는 `이 파일은 "중심 테이블"이다.`

**영어 변수 병기**

- 나쁜 예: `semantic_category는 11종이다.`
- 좋은 예: `semantic_category(대분류)는 11종이다.`

**폴더 구조 (통합 vs 분할)**

- 나쁜 예: `disjoint/` 트리, `nondisjoint/` 트리, `data/` 트리를 각각 따로 3개 코드블록으로 나눠 제시.
- 좋은 예: 최상위부터 하위까지 **하나의 트리**로 통합해 한 코드블록에 제시.

**출처 근거**

- 나쁜 예: `Confluence 페이지에 261,058개 아이템이라고 적혀 있다.` (재인용만)
- 좋은 예: `aws s3 ls --recursive --summarize로 오브젝트 39개·약 4.30GB 실측. categories.csv를 직접 내려받아 3컬럼 구조 확인.`
