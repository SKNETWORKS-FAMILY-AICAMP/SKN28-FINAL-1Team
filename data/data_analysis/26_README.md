# 26 · K-Fashion 이미지·라벨링데이터

## 1. 이 데이터는 무엇인가

`s3://skn28-cozy/26_K_Fashion/`은 한국콘텐츠진흥원(KOCCA)이 AI Hub를 통해 개방한 **K-Fashion 이미지 데이터셋**이다. 실제 패션 이미지(JPG)와 이미지별 라벨(JSON)로 이루어진 국내 최대급 한국 패션 라벨 데이터로, 한 장의 사진 안에서 아우터·상의·하의·원피스를 위치까지 분리해 두고 각 아이템의 속성을 정규화된 값으로 라벨링해 두었다.

데이터 형태는 크게 두 가지다.

- **이미지(JPG)**: 실제 착장 사진 328,139장.
- **라벨(JSON)**: 이미지별 정답지 421,304개. 각 라벨에는 bbox(경계상자) 렉트좌표 + polygon(폴리곤) segmentation(분할) 좌표 + 아이템별 상세 속성(기장/색상/카테고리/소재/프린트/핏 등)이 들어 있다.

우리 프로젝트와의 관계: 아이템 단위 추천·속성 매칭에 즉시 투입 가능한 데이터다. 아이템마다 9~10개 속성이 정규화 값으로 라벨링돼 있어 별도 어노테이션 없이 룰 추출·속성 필터가 가능하고, bbox·polygon으로 위치까지 있어 아이템 단위 crop·검색에 바로 쓸 수 있다. 다만 사용자별 선호도나 신체 치수는 없어 체형 추천(RAG 1)에는 별도 데이터 결합이 필요하다.

| 주의 — 파일명 특수문자 | 원본 `이미지 파일명`에 괄호·한글이 섞여 있어 파일 경로 처리 주의 |

장점 요약: 아이템별 정규화 속성(9~10종) + bbox와 polygon 동시 제공(검출·분할 양쪽 사용 가능) + 스타일 23종·서브스타일 계층 + Training/Validation 분리가 명확하다.

## 4. 가장 중요한 파일

이 데이터셋의 중심축은 **`라벨링데이터/<스타일>/<이미지ID>.json`** 라벨 파일이다. 이미지 1장의 위치(bbox·polygon)와 아이템 속성이 모두 정규화되어 이 한 파일에 담긴다. `이미지 정보.이미지 식별자`(이미지 ID)로 같은 이름의 JPG와 1:1 연결된다.

주요 키:

| 키 | 의미 |
|---|---|
| `이미지 정보.이미지 식별자` | 이미지 ID(식별자) — 원본 JPG와 조인하는 키 |
| `이미지 정보.이미지 높이/너비` | 이미지 크기(픽셀) — bbox·polygon 좌표 정규화 기준 |
| `데이터셋 정보.렉트좌표.<슬롯>` | bbox(경계상자): `X좌표`/`Y좌표`/`가로`/`세로` |
| `데이터셋 정보.폴리곤좌표.<슬롯>` | polygon(폴리곤) segmentation(분할) 좌표 |
| `데이터셋 정보.라벨링.스타일` | 이미지 최상위 스타일 라벨 (예: 섹시) |
| `데이터셋 정보.라벨링.<슬롯>` | 아이템별 속성: `색상`/`카테고리`/`소매기장`/`소재`/`핏` 등 |

`<슬롯>`(slot)은 `아우터`/`상의`/`하의`/`원피스` 4종이며, 착장한 슬롯만 값이 차고 나머지는 빈 객체 `{}`로 남는다.

실제 레코드 발췌(`998905.json`) — 원피스 슬롯만 채워진 사례:

```json
{
  "이미지 정보": {"이미지 식별자": 998905, "이미지 높이": 800, "이미지 너비": 800, "이미지 파일명": "..."},
  "데이터셋 정보": {
    "파일 생성일자": "2020-11-10 15:08:45",
    "데이터셋 상세설명": {
      "렉트좌표": {"원피스": [{"X좌표": 269.2, "Y좌표": 52.5, "가로": 315, "세로": 627}], "아우터": [{}], "상의": [{}], "하의": [{}]},
      "폴리곤좌표": {"원피스": [{"X좌표1": "...", "Y좌표1": "..."}]},
      "라벨링": {
        "스타일": [{"스타일": "섹시"}],
        "원피스": [{"색상": "핑크", "카테고리": "드레스", "소매기장": "민소매", "소재": ["저지"], "핏": "타이트"}]
      }
    }
  }
}
```

## 5. 이 데이터로 알 수 있는 것

| 키워드 | 설명 (근거 파일/변수) |
|---|---|
| 스타일 분포 | 스타일 23종(모던/로맨틱/섹시 등)별 자주 쓰이는 카테고리·색상·소재·핏 분포 (`라벨링.스타일` × `라벨링.<슬롯>` 집계) |
| 아이템 위치 | 한 사진 안에서 아우터/상의/하의/원피스가 각각 어디에 있는지 (`렉트좌표` bbox + `폴리곤좌표` polygon) |
| 아이템 속성 | 아이템별 정규화 속성. 예: 상의 = 티셔츠 / 라운드넥 / 긴팔 / 우븐 / 타이트 (`라벨링.<슬롯>`) |
| 스타일 계층 | 스타일 ↔ 서브스타일 계층. 예: 스타일 "모던" 안에 서브스타일 "매니시" (`라벨링.스타일`) |
| 검출·분할 학습 | 이미지 + segmentation 마스크를 함께 쓰는 의류 검출/분할 모델 학습 근거 (JPG + `폴리곤좌표`) |

자동화 추천 관점에서의 RAG별 활용 강도:

| RAG | 강도 | 데이터 근거 |
|---|---|---|
| RAG 2 (패션 아이템) | 매우 강함 | 아이템마다 9~10개 속성이 정규화 값으로 라벨링 + bbox·polygon 위치 → 아이템 단위 검색·crop·속성 매칭이 라벨링 비용 0으로 가능 |
| RAG 3 (TPO/스타일) | 강함 | 23개 스타일 + 스타일↔서브스타일 계층, 폴더·JSON 이중 라벨 → 스타일별 아이템·색상·핏 분포 근거 |
| RAG 1 (체형/사이즈) | 약함 | 신체 치수 정량 데이터 없음. polygon 면적·bbox 세로/가로 비율로 실루엣 추정만 가능 |

이 데이터로 **알 수 없는 것**:

- 사용자별 선호도 — 개인화 추천용 user-item interaction(사용자-아이템 상호작용) 없음
- 브랜드·가격·재고·구매 정보 — 순수 이미지 + 스타일/속성 라벨만 존재
- 명시적 TPO(출근/여행/데이트 등) 라벨 — 스타일 카테고리로만 간접 추론
- 시간·계절·촬영연도 트렌드 — 파일 생성일자만 있고 트렌드 시계열 없음
- 사용자 리뷰·만족도 텍스트 — 라벨은 속성 정답지뿐
- 신체 치수 / 사이즈 — 사진 속 인물 실루엣만 존재

## 6. 추천 시스템에 활용하는 방법

### 6.1 아이템 속성 룰 추출 (RAG 2)

라벨링 JSON `라벨링.<슬롯>[].{색상, 카테고리, 소재, 핏}`이 모두 정규화 값이라 룰 추출이 라벨링 비용 0이다. Training 라벨 356,467 JSON을 스타일별로 `(카테고리, 색상, 소재, 핏)` 조합 빈도로 집계 → `style_item_attr_freq` 테이블 생성. 예: ("섹시","드레스","핑크","저지","타이트", 812) → "스타일=섹시 && 핏=타이트" 추천 가중치 상향 같은 룰 엔진 입력.

### 6.2 아이템 단위 이미지 검색 (RAG 2)

`폴리곤좌표`를 마스크로 변환해 아이템 영역만 crop(배경 제거) → FashionCLIP 또는 CLIP-ViT-L/14로 512-d 임베딩 → 벡터DB(Pinecone/Weaviate)에 (스타일, 카테고리, 색상) 메타와 함께 저장 → 사용자 옷 사진 1장 → 동일 파이프라인 → top-K 유사 아이템 검색. polygon이 segmentation 마스크로 즉시 변환되므로 별도 어노테이션이 불필요하다. (Training 데이터 한정 — Validation은 polygon이 생략된 JSON이 다수)

### 6.3 스타일 분류 baseline (RAG 3)

JPG 입력 → EfficientNet-B3 또는 ConvNeXt-Tiny로 23-class 분류 → softmax 확률을 ranking score의 한 축으로 사용. 학습은 `라벨링.스타일[0].스타일` 라벨 사용. 주의: Training 라벨은 10개 대표 스타일, Validation은 23개 전체이므로 23-class 통합 매핑 후 동일 매핑으로 평가해야 비교 가능.

### 6.4 속성 필터 검색

정규화 속성(색상/소매기장/넥라인/프린트 등)을 그대로 메타 필터로 사용 → "스타일=섹시 + 색상 핑크/레드 + 핏 타이트 + 카테고리 드레스" 같은 조건으로 즉시 hard filter. 추천 파이프라인의 후보 축소 단계에 바로 투입.

### 6.5 체형 보완 (RAG 1)

신체 치수는 없지만 polygon 면적/세로-가로 비율로 실루엣 특징을 2차 추정할 수 있다(area 작고 ratio 크면 마른 체형 가설 등). 정확도는 낮으며 정밀 체형 추천은 10·20·22번 데이터셋 결합이 필요하다.

### 6.6 통합 아키텍처 (3 RAGs)

```text
              사용자 입력 (체형·선호 스타일·TPO·날씨)
                          │
        ┌─────────────────┼─────────────────┐
        ▼                 ▼                 ▼
   RAG 1 (체형)      RAG 2 (아이템)     RAG 3 (스타일/TPO)
   10·20번 데이터   26번·01번 데이터   26번 스타일 JSON
   키·몸무게·치수   bbox·polygon·속성   23종 + 서브스타일
        │                 │                 │
        └─────────────────┼─────────────────┘
                          ▼
              후보 아이템 100~200개 (hard filter)
                          │
                          ▼
              LLM reranker (이 프로젝트의 LLM 추천)
                          │
                          ▼
              최종 추천 top-5 + 코디 설명
```

### 6.7 우선순위 액션 (이 데이터 한정)

| 순서 | 액션 | 데이터 상태 | ROI |
|---|---|---|---|
| 1 | 스타일 × 아이템 속성 빈도 테이블 생성 | 356,467 JSON, 라벨링 비용 0 | 매우 높음 |
| 2 | bbox·polygon crop → CLIP 임베딩 인덱스 | 23 스타일 polygon 활용 가능 | 높음 |
| 3 | 스타일 23-class 분류기 학습 | Training 10종 / Validation 23종 (불일치 해소 먼저) | 중간 |
| 4 | polygon 비율 기반 실루엣 특징 추정 | 즉시 가능 | 낮음 (보조 feature) |
| 5 | 다른 RAG(10·20·22·23번)와 join 키 설계 | `이미지 식별자` PK 확보 후 | 높음 |

먼저 쓸 것은 라벨링 JSON을 파싱한 (스타일 × 아이템 × 속성) 빈도 테이블로 RAG 2(아이템) + RAG 3(스타일)부터 보강하는 것이다. 아이템 속성이 이 데이터의 최대 강점. 확장 전에 이미지↔라벨 개수 불일치와 Training/Validation 스타일 집합 차이를 ID 매칭 스크립트로 먼저 정리해야 통계가 틀리지 않는다.

## 7. 샘플 보는 순서 + 최소 분석 루틴

`998905.jpg` + `998905.json` 한 쌍을 열었을 때 추출할 정보를 단계별로:

1. **이미지 정보 먼저 읽기** — `이미지 식별자`, `이미지 높이/너비`, `이미지 파일명`.
   *왜?* 좌표값(bbox·polygon)이 절대 픽셀 좌표라서, 이미지 크기(800×800)를 알아야 좌표를 정규화(0~1)해 다른 해상도로 리사이즈할 때 마스크가 맞는다.
2. **라벨링 → 스타일 확인** — `라벨링.스타일 = [{스타일: 섹시}]` (샘플은 서브스타일 없음).
   *왜?* 스타일이 이 이미지의 최상위 정답 라벨. 폴더명(섹시)과 JSON 값이 일치하는지 교차검증하는 기준점.
3. **어떤 아이템 슬롯이 채워졌는지 확인** — 아우터/상의/하의/원피스 중 값이 있는 슬롯만 실제 착장.
   *왜?* 샘플은 원피스만 채워지고 나머지는 빈 `{}`. 빈 슬롯을 "없음"으로 처리해야 아이템 개수 통계가 틀리지 않는다.
4. **아이템 속성 파싱** — 원피스 = 색상 핑크 / 카테고리 드레스 / 소매기장 민소매 / 소재 저지 / 핏 타이트.
   *왜?* 속성이 정규화된 카테고리 값이라 라벨링 없이 바로 속성별 필터/추천 룰(예: "핑크 + 타이트 드레스")로 쓸 수 있다.
5. **좌표(렉트/폴리곤) 마지막에 보기** — bbox는 crop용, polygon은 segmentation 마스크용.
   *왜?* 아이템 단위 crop → 임베딩 → 유사 아이템 검색 파이프라인의 입력. 속성 확인이 끝난 뒤 좌표를 봐야 "무엇을 crop할지"가 정해진다.
