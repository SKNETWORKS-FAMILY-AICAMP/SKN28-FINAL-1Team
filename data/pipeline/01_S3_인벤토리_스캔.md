# 워크플로 1 — S3 인벤토리 스캔

`s3://skn28-cozy`를 매일 스캔해 이전 결과와 비교하고, 신규/변경 파일이 있으면 정제 워크플로를 자동으로 트리거한다.

실물: `data/pipeline/01_workflow_s3_scan.json` (n8n 워크플로 ID: `WORKFLOW_ID_SCAN`)
노드 수: 7개

---

## 1. 무엇을 하는 워크플로인가

- 파이썬(`s3_scan.py`)이 `aws s3 ls --recursive` 결과를 이전 스캔과 diff한다. [근거: scripts/s3_scan.py]
- 신규·변경 파일이 있으면 Slack 알림 + 워크플로 2(정제) 트리거.
- 없으면 종료.

---

## 2. 노드 흐름도

```text
[Schedule Trigger]              scheduleTrigger, cron "0 9 * * *" (매일 09:00 KST)
      │
      ▼
[Run S3 Scanner]                executeCommand
      │                         python3 /repo/data/pipeline/01_s3_scan.py
      │                         --output /tmp/s3_inventory.json
      │                         --state-file /home/node/.n8n/s3_previous_inventory.json
      ▼
[Diff with Previous Scan]       code (JS)
      │                         /tmp/s3_inventory.json 읽어 diff 파싱
      │                         → new/modified 키 목록·건수 산출, has_changes 판정
      ▼
[If Changes Exist]              if  (has_changes == true?)
      ├─ true ─┬─▶ [Slack: Notify Changes]   slack, #fashion-data-alerts (continueOnFail)
      │        └─▶ [Trigger Cleaning Workflow] executeWorkflow → WORKFLOW_ID_CLEAN
      └─ false ──▶ [No Changes - Skip]        noOp (종료)
```

노드 7개: Schedule Trigger, Run S3 Scanner, Diff with Previous Scan, If Changes Exist, Slack: Notify Changes, Trigger Cleaning Workflow, No Changes - Skip. [근거: workflow_01_s3_scan.json]

---

## 3. 왜 이렇게 설계했는가

- **스캔·diff는 파이썬이 한다.** `aws s3 ls` 호출, 파싱, 이전 상태와 diff는 전부 `s3_scan.py`가 처리한다. n8n은 스크립트를 부르고 결과 파일(`/tmp/s3_inventory.json`)만 Code 노드로 읽는다. [근거: scripts/s3_scan.py, workflow JSON]
- **상태 파일로 증분 감지.** 이전 스캔 결과를 `/home/node/.n8n/s3_previous_inventory.json`(named volume)에 저장해, 재기동 후에도 diff 기준이 유지된다. [근거: workflow JSON command, docker-compose.yml n8n_data]
- **n8n은 분기·알림·다음 단계 호출만.** IF 노드로 변경 유무를 판정하고, 있으면 Slack과 워크플로 2를 병렬로 호출한다. 무거운 로직이 노드에 없다. [근거: n8n/README.md §정직한 평가]

---

## 4. 실행 방법

| 방식 | 내용 |
|---|---|
| 자동 (트리거) | Schedule Trigger, 매일 09:00 KST. **현재 비활성** — 수동 검증 후 활성화 예정 |
| 수동 | n8n에서 워크플로 열고 우측 상단 Execute Workflow → S3 리스트/diff/Slack까지 도는지 확인 [근거: n8n/README.md §5] |

점검 명령 (Execute Command 실패 시):
```powershell
docker compose exec n8n python3 --version
docker compose exec n8n ls /repo/data/pipeline/scripts
```

전제: `AWS_SECRET_ACCESS_KEY`가 Infisical(env=dev)에 등록돼야 `aws s3 ls`가 성공한다. **현재 미등록 → S3 접근 실패 상태.**

## 5. 관련 파일 경로

| 종류 | 경로 |
|---|---|
| 워크플로 JSON | `data/pipeline/01_workflow_s3_scan.json` |
| 스캔 스크립트 | `data/pipeline/01_s3_scan.py` |
| 결과 파일(런타임) | `/tmp/s3_inventory.json` (컨테이너 내부) |
| 상태 파일(런타임) | `/home/node/.n8n/s3_previous_inventory.json` (named volume) |

주의: Execute Command 노드가 `python3 /repo/data/pipeline/01_s3_scan.py`를 참조한다. compose가 레포 루트를 `/repo`로 마운트하므로 스크립트를 `data/pipeline/` 밖으로 옮기면 워크플로가 깨진다. [근거: docker-compose.yml volumes `../../..:/repo:ro`]
