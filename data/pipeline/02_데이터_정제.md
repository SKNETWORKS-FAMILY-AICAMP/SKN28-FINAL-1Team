# 워크플로 2 — 데이터 정제

신규/변경 파일을 EUC-KR/CP949 → UTF-8로 인코딩 정제하고 JSON 스키마를 검증한다. 실패가 없으면 임베딩 워크플로를 트리거한다.

실물: `data/pipeline/02_workflow_cleaning.json` (n8n 워크플로 ID: `WORKFLOW_ID_CLEAN`)
노드 수: 9개

---

## 1. 무엇을 하는 워크플로인가

- 파이썬(`clean_encoding.py`)이 텍스트/JSON 파일을 UTF-8(BOM 제거)로 변환한다. S3 다운로드/재업로드까지 스크립트가 담당. [근거: scripts/clean_encoding.py]
- 파이썬(`validate_json.py`)이 정제된 JSON의 스키마·필수 필드를 검증한다. [근거: scripts/validate_json.py]
- 실패 없으면 워크플로 3(임베딩) 트리거, 있으면 Slack 실패 알림.

---

## 2. 노드 흐름도

```text
[Webhook Trigger]              webhook, POST /clean
      │
      ▼
[Prepare Batch Files]          code (JS)
      │                        입력 키 정규화 → 텍스트/이미지 키 분리
      │                        텍스트 키를 /tmp/clean_batch.json 에 기록
      ▼
[Run Encoding Cleaner]         executeCommand (continueOnFail)
      │                        python3 .../clean_encoding.py --batch /tmp/clean_batch.json
      ▼
[Prepare Validation Batch]     code (JS)
      │                        정제 stdout 파싱 → 성공한 로컬 .json 경로만 추려
      │                        /tmp/validate_batch.json 에 기록, 정제 오류 집계
      ▼
[Validate JSON Schemas]        executeCommand (continueOnFail)
      │                        python3 .../validate_json.py --batch /tmp/validate_batch.json
      ▼
[Parse Cleaning Results]       code (JS)
      │                        검증 stdout 파싱 → 정제+검증 실패 합산, has_failures 판정
      ▼
[If Cleaning Success]          if  (has_failures == false?)
      ├─ true  ──▶ [Trigger Embedding Workflow]  executeWorkflow → WORKFLOW_ID_EMBED
      └─ false ──▶ [Slack: Cleaning Failed]       slack, #fashion-data-alerts (continueOnFail)
```

노드 9개: Webhook Trigger, Prepare Batch Files, Run Encoding Cleaner, Prepare Validation Batch, Validate JSON Schemas, Parse Cleaning Results, If Cleaning Success, Trigger Embedding Workflow, Slack: Cleaning Failed. [근거: workflow_02_cleaning.json]

주의: JSON의 노드 `position` 좌표와 실제 실행 순서는 다르다. 위 흐름도는 `connections` 블록(실제 연결)을 기준으로 그린 것이다. [근거: workflow_02_cleaning.json connections]

---

## 3. 왜 이렇게 설계했는가

- **정제·검증·S3 입출력은 전부 파이썬.** 인코딩 감지(utf-8 → utf-8-sig → cp949 → euc-kr → latin-1 순 시도), BOM 제거, S3 다운로드/업로드를 `clean_encoding.py`가 처리한다. n8n은 배치 파일만 만들어 넘긴다. [근거: scripts/clean_encoding.py FALLBACK_ENCODINGS]
- **Code 노드는 "얇은 접착제".** 입력 키 정규화, 배치 파일 쓰기(`/tmp/*.json`), stdout JSON 파싱 같은 가벼운 변환만 한다. Node.js `fs` 모듈만 허용된다. [근거: docker-compose.yml `NODE_FUNCTION_ALLOW_BUILTIN=fs`]
- **continueOnFail로 실패를 삼키지 않고 집계.** Execute Command 노드가 죽어도 흐름을 멈추지 않고, 다음 Code 노드가 exitCode·error를 읽어 실패 건수로 집계한다. 그래야 IF 노드가 실패 여부를 판정하고 Slack으로 알릴 수 있다. [근거: workflow JSON continueOnFail, code-parse jsCode]
- **텍스트/이미지 분리.** 정제는 텍스트 파일만 대상이고, 이미지 키는 그대로 보존해 워크플로 3(임베딩)으로 넘긴다. [근거: Prepare Batch Files jsCode의 textExts/imageExts]

## 4. 실행 방법

| 방식 | 내용 |
|---|---|
| 자동 (트리거) | 워크플로 1의 Trigger Cleaning Workflow가 호출 (executeWorkflow). Webhook `POST /clean`도 존재 |
| 수동 | 각 노드를 Execute step으로 하나씩 눌러 `python3 /repo/...`가 컨테이너 안에서 도는지 확인 [근거: n8n/README.md §5] |

전제: `AWS_SECRET_ACCESS_KEY` 등록돼야 S3 다운로드/업로드 성공. **현재 미등록 → S3 접근 실패 상태.**

## 5. 관련 파일 경로

| 종류 | 경로 |

| 워크플로 JSON | `data/pipeline/02_workflow_cleaning.json` |
