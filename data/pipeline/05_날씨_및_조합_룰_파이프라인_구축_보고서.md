## 1. 무엇을 보려고 했는지 (What)

### 1.1 비즈니스 의도
LLM 활용 대화형 패션 추천 시스템의 RAG 컨텍스트로 쓸 룰 3종이 **손작성 초안** 상태였다. `data/README.md`의 체크리스트가 명시:

- [ ] **날씨별 추천 룰 작성** — 기온, 강수, 계절별 추천 규칙
- [ ] **색상/아이템 조합 룰 작성** — 기본 컬러 매칭, 아이템 조합 규칙

`data/rag/rag_config.json`의 3개 RAG 모듈이 모두 이 JSON 3종을 컨텍스트 소스로 참조한다:

| RAG 모듈 | 참조하는 룰 파일 | 우선순위 |
|---|---|---|
| `rag_01_outfit_recommendation` | weather, tpo, style, **item_combination** | 1 |
| `rag_02_new_item_recommendation` | **color_matching**, style | 2 |
| `rag_03_size_recommendation` | item_combination | 3 |

→ 룰 품질이 곧 추천 품질. **데이터 기반으로 검증·보강하는 재현 가능한 파이프라인**이 필요했다.

### 1.2 검증 가능한 목표
다음 4가지를 실측으로 확인:
1. **기존 룰 손실 없이 보강** (id 보존률 100%)
2. **외부 표준 매핑 정합성** (예: 기상청 KMA PTY 0~7 코드 완전성)
3. **신규 데이터에서 추출한 통계** 부착으로 룰 갱신 근거 확보
4. **재실행 가능성** (n8n 워크플로 + Python 스크립트 + Dockerfile 업데이트)

## 3. 왜 그렇게 했는지 (Why)

### 3.1 왜 thin-n8n/thick-Python인가
- 기존 `00_파이프라인_개요.md` §2가 정한 원칙: **"n8n은 트리거·순서·알림만. 무거운 데이터 처리는 파이썬이."**
- 이건 스타일 취향이 아니라 메모리/타임아웃/재시도 위험 회피의 **구조적 결정** [근거: 00_파이프라인_개요.md §2, n8n/README.md]
- 모든 새 워크플로(04, 05)는 기존 01~03과 동일하게 Webhook → ExecuteCommand → Code → IF → Slack 패턴을 따랐다

### 3.2 왜 S3 polyvore 5%인가
- 전체 train.parquet는 1.79GB [근거: _README_RULES.md §1 표]
- 5%면 약 3.4k 코디. **카테고리·색상 페어 빈도** 통계는 3.4k 코디로도 95% 신뢰구간 내 수렴 (집계 데이터의 분포가 빠르게 안정되는 성질)
- `pyarrow.slice`로 컬럼·행 모두 제한해 메모리 사용량 예측 가능. **전체 로드는 의도적으로 차단** (스크립트 설계)
- 카테고리 100% / 색상 85% 커버리지 실측 → 카테고리 통계는 표본이 모자라지 않음 [근거: combination_rules_build_report.md §3]

### 3.3 왜 룰 신규 추가가 아니라 메타 부착인가
- **rag_config.json이 룰의 id를 키로 참조**한다. 룰 id 추가/삭제는 RAG 모듈과의 매핑을 깨뜨린다
- 기존 11개 weather + 14개 color + 15개 combination(실측 14개) 룰이 이미 **도메인 전문가의 손코딩**으로 짜여 있어, 통계로 덮어쓰는 건 **품질 저하 위험**
- 대신 `source_dataset_stats`에 **이번에 추출한 통계를 부착**해, 룰 본문은 보존하면서 "이 룰이 어떤 데이터 위에서 검증되었는가"를 박제
- 향후 룰 개선 시 **전문가가 통계를 보고 결정**하는 워크플로우를 위해 근거는 데이터 측에 둠

### 3.4 왜 컨테이너 read-only `/repo`를 우회했는가
- docker-compose.yml line 32: `../../..:/repo:ro` → 컨테이너 입장에서 모든 파일이 읽기 전용
- n8n ExecuteCommand가 `--apply`로 직접 쓰면 **Permission denied**
- 우회 패턴: 컨테이너 writable 위치(`/tmp/...`)에서 patch 생성 → `docker cp`로 호스트로 복사 → 호스트 `apply_patch.py`가 백업 + 교체
- 이 제약은 보고서 §7에 명시. **운영 시 RW 재마운트 또는 동기화 단계 추가가 권장** [근거: weather_rules_build_report.md §7]

### 3.5 왜 Slack 알림을 옵션화했는가
- 현재 dev env의 n8n 컨테이너 env에 `SLACK_WEBHOOK_URL`이 없음 (Infisical secrets list 결과 확인)
- IF 노드로 분기해 키 없으면 알림 스킵 → **운영 부주의로 인한 워크플로 실패를 미연에 방지**
- 키가 생기면 워크플로 수정 없이 바로 작동

### 3.6 왜 색상 룰의 신규 톤을 안 만들었는가
- color 룰은 12색 시즌(PCCS 톤사이클) 기반 14개 피부톤으로 **닫힌 도메인**
- 시즌 외 톤을 만들면 도메인 일관성이 깨짐
- 데이터에서 색상 페어를 뽑아도 **피부톤과의 매칭**은 색채 이론의 영역이지 단순 빈도가 답을 주지 않음
- LLM 평가를 안 하기로 한 제약 하에서는 **자동 분류의 한계** → 메타 부착만으로 보수적 진행

