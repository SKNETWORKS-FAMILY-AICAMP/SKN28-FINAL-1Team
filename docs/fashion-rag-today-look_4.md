# 오늘의 룩 추천 RAG 계획: S3 데이터 기반 임베딩·리트리버

- **작성일**: 2026-07-14
- **선행 문서**: `fashion-rag-embedding-pipeline_1.md`(인프라), `fashion-rag-embedding-retriever_2.md`(임베딩·리트리버 공통 설계), `fashion-rag-s3-datasets_3.md`(S3 실측 인벤토리), `local/개인맞춤룩추천_기능설계서.md`
- **대상 기능**: **오늘의 룩** — 기온·날씨 + 사용자 체형 + 옷장 데이터로 오늘 입을 코디 조합 추천
- **범위 결정**: 추천 결과는 **사용자 옷장(wardrobe) 아이템만**으로 구성한다. 상품(products) 보완 제안은 본 문서 범위 밖(후속 확장).
- **성격**: 계획 문서. 구현 없음.

---

## 1. 기능 정의와 RAG의 역할

```
입력:  weather_tool {temp_min, temp_max, feels_like, precip_prob, wind, ...}
       profile     {체형 분류, 키·몸무게(·어깨너비 등), 추구 스타일, 피부톤}
       wardrobe    사용자 소유 아이템 (Qdrant wardrobe 컬렉션)
출력:  코디 조합 1~3개 (상의+하의±아우터±신발, 각 아이템은 옷장 소유물)
       + 근거 설명 ("오늘 최저 3℃라 아우터 필수, 이 체형에는 ~핏이 유리" 류)
```

핵심 관찰 두 가지가 설계를 결정한다.

1. **후보 풀이 작다.** 옷장은 수십~수백 아이템 규모다. products처럼 "수십만 중 벡터 검색으로 좁히는" 문제가 아니라, **작은 풀에서 좋은 "조합"을 고르고 그 근거를 대는** 문제다. 따라서 RAG의 무게중심은 후보 검색이 아니라 ① 조합의 정답 사례(어떤 옷들이 실제로 같이 입히는가), ② 날씨·체형 판단의 정량 근거에 있다.
2. **사이즈 적합성은 이 기능에서 부차적이다.** 옷장 아이템은 이미 소유한(= 몸에 맞는) 옷이다. 체형 데이터의 역할은 사이즈 판정이 아니라 **실루엣 권장/기피**(체형별 유리한 핏·기장)와 설명 근거다.

이에 따라 S3 원본 데이터는 세 용도로만 쓴다: **(A) 코디 조합 사례의 벡터 검색**(looks), **(B) 지식 청크·규칙 상수의 집계 생산**(knowledge + 코드 상수), **(C) 평가셋**. S3 데이터가 추천 결과로 사용자에게 직접 노출되는 일은 없다(문서 3의 9절 라이선스 격리 원칙 유지).

## 2. S3 데이터셋 활용 판정

문서 3의 실측 인벤토리에서 이 기능에 쓰는 것만 추린다.

| S3 데이터셋 | 용도 | 벡터 임베딩 여부 | 우선순위 |
|---|---|---|---|
| `26_K_Fashion/` | **(A)** 룩 조합 사례 → looks 컬렉션. **(B)** 공출현 통계 → 조합 스코어·Rule Validator 상수, 스타일 프로파일 청크 | **O** (이미지+직렬화 텍스트) | **P0** |
| `01_의류통합데이터/` | **(B)** `metadata.clothes.season`·소재·소매기장 집계 → **기온↔소재·복종 매핑의 정량 근거** (filter_rules 상수 + 날씨 지식 청크). 2차: wear컷+모델 신체치수 → `fit_examples` 컬렉션("나와 비슷한 체형의 착장 사례") | 1차 **X**(집계만) / 2차 O | P1 |
| `22.사이즈코리아/` | **(B)** 성별·연령 퍼센타일 → body_classifier 상수 + 체형 분포 지식 청크 | X (산출 청크만 임베딩) | P1 |
| `11. ETRI 패션 코디/` | **(C)** (요청 발화 → 정답 속성) 평가 골든셋, Stylist few-shot | X | P2 |
| `05_polyvore/` | **(B)** 조합 스코어러 학습·평가 (Rule Validator 뒤의 2차 검증) | X (모델 학습용) | P2 |
| `23_연도별/`, `03/10/12/20`, H&M | 이 기능 비대상 (문서 3 판정 유지) | — | — |

> 판정 원칙: **"날씨 적합성·체형 적합성은 벡터로 풀지 않는다."** 두 조건 모두 위반 시 추천이 틀리는 하드 조건(한겨울에 린넨 반팔)이거나 정량 규칙(퍼센타일)이므로, 문서 2의 3-2절 필터 정책대로 **payload 필터 + 코드 상수**로 처리한다. 벡터 임베딩이 담당하는 것은 "스타일 무드가 비슷한 룩 찾기"뿐이다. 따라서 01·22는 임베딩 대상이 아니라 **집계 원천**이다.

## 3. 임베딩 계획

컬렉션 4종(문서 3의 체제)을 그대로 쓰되, 이 기능이 요구하는 payload·청크를 보강한다. 모델은 문서 2 확정안 재사용: 이미지 = Marqo-FashionSigLIP(768), 텍스트 = BGE-M3(1024), named vector.

### 3-1. wardrobe — 추천 후보 (문서 2의 2-5절 보강)

옷장 아이템은 업로드 시 단건 임베딩(이미지 벡터 + Vision LLM 태깅 → 직렬화 텍스트 벡터)한다. 이 기능을 위해 **태깅 스키마에 날씨 판단용 파생 필드를 추가**한다.

```
payload 추가 (Vision LLM 태깅 시 함께 산출):
  warmth_level: 1~5        # 소재·소매기장·기장·복종에서 유도. 기온 필터의 축
  rain_friendly: bool      # 스웨이드·니트 등 감점 소재 여부
  layer_role: keyword      # inner/top/bottom/outer/shoes — 슬롯 구성의 전제 (기존)
```

- `warmth_level` 산출 규칙은 LLM 자유 판단이 아니라 **01 의류통합 집계로 만든 매핑 테이블**(소재×복종×소매기장 → 등급)을 프롬프트에 상수로 제공해 일관성을 확보한다. 기능설계서 5-1절의 `warmth_level`·`rain_friendly` 필드 정의를 그대로 따른다.
- 기존 태깅 완료분은 payload만 재산출(재임베딩 불필요 — 벡터 입력 문장은 불변)하는 backfill 스크립트로 처리한다.

### 3-2. looks — 코디 조합 사례 (문서 3의 4절 재사용 + 날씨 프록시 보강)

K-Fashion 라벨에는 계절·기온 정보가 없다. 대신 착장 속성(소매기장·기장·소재·아우터 유무)이 계절의 프록시다. 문서 3의 looks 스키마에 파생 payload를 추가한다.

```
payload 추가 (적재 시 라벨에서 유도):
  warmth_proxy: 1~5        # 파트별 소재·소매기장·아우터 유무 → 룩 전체 보온 등급
  has_outer: bool          # 아우터 착장 여부 — 기온별 룩 필터의 1차 축
```

- 유도 규칙은 wardrobe의 `warmth_level`과 **같은 매핑 테이블**을 쓴다(`ml/common/`에 공용 상수). 그래야 "오늘 기온에 맞는 참조 룩"과 "오늘 기온에 맞는 옷장 아이템"이 같은 언어로 필터된다.
- 텍스트 벡터는 문서 3의 `serialize_look` 그대로. 이미지 벡터(전신 룩 컷)는 그대로 — 옷장 아이템 이미지와의 직접 유사도 비교에는 쓰지 않고(전신 vs 단품 도메인 갭), 무드 검색용으로만 쓴다.

### 3-3. knowledge — 날씨·체형 지식 청크 (집계 생산, 문서 3의 5절 흐름)

이 기능이 소비하는 청크 3종을 S3 집계로 생산한다. 산출 md는 `docs/knowledge/generated/`에 커밋(사람 리뷰 가능), BGE-M3로 임베딩.

| 청크 유형 | 원천 | 내용 예 | payload |
|---|---|---|---|
| 날씨-의상 매핑 | 01 의류통합 (`clothes.season`×소재×복종 집계) + 수작업 검수 | "일 최고 9~15℃에서는 코트·니트 조합이 일반적, 린넨·시어서커는 여름 소재로 부적합" | `{knowledge_type: "weather", temp_band, season}` |
| 체형 실루엣 가이드 | 사이즈코리아 퍼센타일 + 수작업 가이드(기능설계서 5-2) | "어깨너비 상위 20% 체형은 상의 볼륨을 줄이고 하의에 부피감" | `{knowledge_type: "body", body_type}` |
| 조합 규칙 | K-Fashion 공출현 집계 | "와이드 하의 룩의 상의는 크롭·노멀 기장이 다수" | `{knowledge_type: "combination", style}` |

**이중 원천 방지 원칙(문서 2의 3-3① 유지)**: 기온 구간 → 필터 변환, warmth 매핑, 체형 → 핏 권장은 전부 **코드 상수(`filter_rules.py`)가 단일 판단 원천**이다. knowledge 청크는 판단하지 않고 **Stylist의 설명 근거 검색용**으로만 쓴다. 집계 스크립트가 상수와 청크를 같은 원천에서 동시 생성하므로 수치 불일치가 구조적으로 방지된다.

### 3-4. fit_examples — 체형별 착장 사례 (2차, 도입 보류)

01 의류통합의 wear컷을 FashionSigLIP로 임베딩하고 모델 신체치수를 payload(range 인덱스)로 붙이면 "키 160·어깨 41 체형의 착장 사례" 검색이 가능하다(문서 3의 5-3절). 단, 오늘의 룩 1차 응답에는 체형 지식 **청크**만으로 근거가 충분할 것으로 보고, **P1 청크의 효과 평가 후 도입을 판단**한다(문서 3의 미확정 5번과 동일 결정 지점).

## 4. 리트리버 계획

문서 2의 공통 파이프라인(필터 빌더 → 하드 필터 → 벡터 랭킹 → 슬롯별 후보)과 문서 3의 2단 검색(looks 경유)을 이 기능에 맞게 조립한다. 리트리버는 LLM 없는 순수 함수형 Tool(기존 원칙).

### 4-1. 전체 흐름

```
① 입력 정형화 (Orchestrator)
   weather_tool + get_profile 병렬 호출 (기능설계서 4.1)
② 필터 빌드 (filter_rules.py — 코드 상수)
   기온 → 슬롯 구성(아우터 필수/생략) + warmth_level range (예: 최저 3℃ → outer 필수, warmth 4~5)
   강수확률 ≥ 60% → must_not: rain_friendly=false 소재
   체형 → should: 권장 fit/silhouette (하드 아님 — 후보 고갈 방지, 문서 2의 3-2절)
   프로필 기피(색·핏) → must_not (끝까지 유지)
③ 참조 룩 검색 (looks, M≈10)
   query_text = 추구 스타일 + 계절 직렬화 ("미니멀 스타일, 겨울 데일리룩")
   filter = warmth_proxy range + has_outer + style should(태그 매핑 4-3절 경유)
④ 룩 분해 (payload 집계, LLM 없음)
   상위 룩들의 parts에서 파트별 카테고리·핏·색 분포 추출 → 슬롯별 should 보강
   (예: 겨울 미니멀 룩 상위에서 "하의: 슬랙스·와이드 다수" → bottom 슬롯 should)
⑤ wardrobe 슬롯별 후보 (핵심 검색)
   must: user_id + layer_role + warmth_level range
   must_not: 기피 색·핏, 강수 부적합 소재
   should: ④의 보강 + 체형 권장 핏
   벡터: ③의 query_text 임베딩 (스타일 무드 정렬용)
   → 슬롯별 상위 K(옷장이 작으므로 K=5~10, 미달 시 완화 사다리: should 제거 → style 완화. warmth·must_not은 유지)
⑥ 조합 생성·검증
   슬롯 후보 교차 → 조합 스코어 = 벡터 점수 + K-Fashion 공출현 점수(색·핏 쌍 빈도)
   → Rule Validator (warmth 합계 적정, 실루엣 균형(와이드+와이드 지양 — 공출현 통계로 채택/기각), 색 규칙)
   → 상위 1~3 조합
⑦ Stylist (LLM)
   knowledge 검색 (weather/body/combination 청크) + 참조 룩 속성 요약
   → "왜 이 조합인가" 설명 생성. 참조 룩 이미지는 내부 근거로만, 응답 노출 금지
```

### 4-2. 옷장 소규모 특성에 따른 판단

- **wardrobe 검색은 벡터보다 필터가 주력이다.** 아이템 수백 개면 필터 후 후보가 한 자릿수인 경우가 흔하다. ⑤에서 후보 < 2인 슬롯은 벡터 랭킹이 무의미하므로 무벡터 scroll(필터만)로 전환하고, 그래도 0이면 해당 슬롯 결측을 Stylist에 전달해 "옷장에 겨울 아우터가 없다"는 응답(향후 상품 보완 제안의 훅)으로 처리한다.
- **조합 폭발 없음.** 슬롯당 ≤10이면 교차 조합도 ≤ 수천 — ⑥의 스코어링을 인메모리로 전량 수행 가능. Qdrant 재질의 불필요.
- **캐시**: 동일 사용자·동일 날짜의 날씨·프로필이 불변이면 ②~④ 결과를 Redis 캐시(TTL 당일). 옷장 변경 시 무효화.

### 4-3. looks → wardrobe 태그 정합

④의 속성 분포를 ⑤의 필터로 옮기려면 K-Fashion enum과 wardrobe 태깅 enum(collector `STYLES` 재사용)이 이어져야 한다. **문서 3의 4-3절 매핑 테이블(`ml/common/tag_schema.py`)을 그대로 사용**한다 — wardrobe가 상품과 동일 태그 체계이므로(문서 2의 2-5절) 추가 매핑은 불필요. 매핑이 근사인 스타일은 should로만 쓴다(기존 원칙).

## 5. 평가 계획

- **골든셋**: (날씨 시나리오 × 체형 × 옷장 구성) 조합 20~30케이스 수작업 구축 + ETRI 대화에서 날씨·TPO 발화 케이스 변환(P2).
- **지표**: 기온 부적합 아이템 노출률(= 0 목표, warmth 필터 검증), 기피 조건 누출률(= 0), 슬롯 결측률, Rule Validator 기각률, 조합 적절성 human eval.
- **A/B**: looks 경유(③④) on/off로 조합 품질 비교 — 옷장 소규모에서 2단 검색이 실익이 있는지 P0 말에 판정.
